# 创建型模式
## 单例模式
<font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">单例模式提供了一种创建对象的最佳方式。</font>

<font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">单例模式涉及到一个单一的类，该类负责创建自己的对象，同时确保只有单个对象被创建。这个类提供了一种访问其唯一的对象的方式，可以直接访问，不需要实例化该类的对象。</font>

+ 意图：<font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">确保一个类只有一个实例，并提供了一个全局访问点来访问该实例。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">主要解决：频繁创建和销毁全局使用的类实例的问题。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">何时使用：当需要控制实例数目，节省系统资源时。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">关键代码：构造函数是私有的。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">优点：内存中只有一个实例，减少内存开销，尤其是频繁创建和销毁实例时。  
</font><font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">	   避免资源的多重占用（如写文件操作）。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">缺点：没有接口，不能继承。  
</font><font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">	   与单一职责原则冲突，一个类应该只关心内部逻辑，而不关心实例化方式。</font>
+ <font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">注意事项：  
</font><font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">	延迟初始化：实例在第一次调用 </font>`<font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">getInstance()</font>`<font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);"> 方法时创建。</font>**<font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">  
</font>****<font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">	</font>**<font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">线程安全：</font>`<font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">getInstance()</font>`<font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);"> 方法中需要使用同步锁 </font>`<font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);">synchronized (Singleton.class)</font>`<font style="color:rgb(51, 51, 51);background-color:rgb(250, 252, 253);"> 防止多线程同时进入造成实例被多次创建。</font>

    // 构造器私有化

    }

/**
 * 懒汉式，一般实现方法

class LazySingleton {
    // 唯一实例
    private static LazySingleton instance;

    // 构造器私有化
    private LazySingleton() {
    }

    // 方法加锁
    public static synchronized LazySingleton getInstance0() {
        if (instance == null) {
            instance = new LazySingleton();
        }
        return instance;
    }

    // 代码块加锁
    public static LazySingleton getInstance1() {
        synchronized (LazySingleton.class) {
            if (instance == null) {
                instance = new LazySingleton();
            }
            return instance;
        }
    }

    // 双检锁/双重校验锁，安全且在多线程情况下能保持高性能。
    public static LazySingleton getSingleton2() {
        if (instance == null) {
            synchronized (LazySingleton.class) {
                if (instance == null) {
                    instance = new LazySingleton();
                }
            }
        }
        return instance;
    }

    // 登记类/静态内部类
    private static class LazySingletonHolder {
        private static final LazySingleton INSTANCE = new LazySingleton();
    }
    public static final LazySingleton getInstance() {
        return LazySingletonHolder.INSTANCE;
    }
}

public class ConfigManager {
    // 私有静态实例，使用volatile确保多线程可见性
    private static volatile ConfigManager instance;

    // 存储配置数据的Map
    private final Map<String, Object> configData;

    // 标记配置是否已加载
    private boolean isLoaded;

    // 私有构造方法，防止外部实例化
    private ConfigManager() {
        configData = new HashMap<>();
        isLoaded = false;
    }

    // 全局访问点，双重检查锁定确保线程安全和懒加载
    public static ConfigManager getInstance() {
        if (instance == null) {
            synchronized (ConfigManager.class) {
                if (instance == null) {
                    instance = new ConfigManager();
                }
            }
        }
        return instance;
    }

    // 测试是否多线程安全
    public static void main(String[] args) throws InterruptedException {
        final int threadCount = 10; // 测试线程数量
        final CountDownLatch startLatch = new CountDownLatch(1); // 控制同时开始
        final CountDownLatch endLatch = new CountDownLatch(threadCount); // 等待所有线程结束
        final List<ConfigManager> instances = new ArrayList<>(); // 存储各线程获取的实例

        // 创建多个线程
        for (int i = 0; i < threadCount; i++) {
            new Thread(() -> {
                try {
                    startLatch.await(); // 阻塞等待，直到主线程发出开始信号

                    ConfigManager instance = ConfigManager.getInstance();
                    synchronized (instances) { // 线程安全地添加实例
                        instances.add(instance);
                    }

                    instance.loadConfig("resource/config.json");
                    System.out.println("key0:" + instance.getConfig("key0"));
                    System.out.println("key1:" + instance.getConfig("key1"));
                    System.out.println("key2:" + instance.getConfig("key2"));

                } catch (InterruptedException e) {
                    e.printStackTrace();
                } finally {
                    endLatch.countDown(); // 线程执行完毕，减少计数
                }
            }).start();
        }

        // 所有线程准备就绪后，发出开始信号
        startLatch.countDown();
        // 等待所有线程执行完毕（最多等待3秒，防止无限阻塞）
        endLatch.await();

        // 验证结果
        System.out.println("===== 多线程实例验证 =====");
        System.out.println("实际收集到的实例数量：" + instances.size());
        System.out.println("所有线程获取的实例是否相同：" + checkAllSame(instances));

        System.out.println("实例哈希值：");
        for (ConfigManager instance : instances) {
            System.out.println(instance.hashCode());
        }
    }

    // 检查所有实例是否为同一个对象
    private static boolean checkAllSame(List<ConfigManager> instances) {
        if (instances.isEmpty()) return false;
        ConfigManager first = instances.get(0);
        for (ConfigManager instance : instances) {
            if (instance != first) return false;
        }
        return true;
    }

    // 加载配置文件
    public void loadConfig(String filePath) {
        if (isLoaded) {
            System.out.println("配置文件已加载，无需重复加载～");
            return;
        }

        System.out.println("从 " + filePath + " 加载配置文件...");
        // 模拟从配置文件中读取配置
        configData.put("key0", "value0");
        configData.put("key1", "value1");
        configData.put("key2", "value2");

        isLoaded = true;
    }

    // 获取配置项
    public Object getConfig(String key) {
        if (!isLoaded) {
            System.out.println("请先加载配置文件！");
            return null;
        }
        return configData.get(key);
    }
}

## 工厂模式
如果需要创建一个对象, 直接new就可以了, 但是工厂方法模式代替了传统的new方法, 那么, 为什么要代替new方法呢?

当我们大量的使用new创建类实例, 如果有一天需要修改类名或者类的构造器, 那么我们就要修改每一处实例!!! 同时一个类实力的创建, 并不是单单的new就够了, 还需要更多步骤来准备构造器的参数, 所以工厂类应运而生!!!

根据迪米特法则, 我们需要尽可能少的与其他类交互, 所以我们可以将new方法封装到一个工厂类中, 调用工厂类中的方法来创建类实例.那么当类出现了变动, 我们也只需要修改工厂类中的方法.

### 实现方式
通过定义顶层抽象工厂类, 然后继承该顶层类, 针对每一个产品都创建对应的工厂类即可

## 抽象工厂模式
当我们一条产品线上有大量的产品时, 工厂方法模式就有些力不从心了, 比如小米有小米手机, 小米智能音响, 小米智能手环等等, 我们就需要为每一个产品创建一个工厂类, 太麻烦!!! 这就是抽象工厂出现的原因.

我们只需要创建一个抽象工厂, 将一条产品线的所有产品都在抽象工厂中生产. 这样我们就只需要针对一条产品线创建一个抽象工厂即可.

缺点: 当新增了产品时, 就不得不在抽象工厂中新增新产品的生产方法, 这就违背了开闭原则!!!

### 实现方式
创建一个工厂类, 这个工厂类中包含了一条产品线上的所有产品的生产方法

## 建造者模式
很多的框架中有xxxBuilder类, 这一般就是使用了建造者模式的类。我们通过建造者不断配置参数或者内容, 当我们配置完所有的参数或内容后, 最后在进行对象的构建。相比于new方法，建造者更加关注如何完成每一步的配置, 同时如果一个类的构造器参数过多, 那么使用建造者也会更加的优雅!!!

### 实现方式
假设, 现在有A类, 以及A的建造者ABuilder类, 那么我们需要将A类的构造器私有化, 保证只能通过Abuilder类进行A类实例的创建;  
	而ABuilder一般需要包含A类中的所有字段, 针对ABuilder中的每一个字段都需要有一个方法对其进行值设置, 这个方法可以返回this, 这样便于使用链式编程;  
	除此之外, ABulider中还需要一个builer方法, 用于创建A类实例, 这里创建A类实例所使用的值就是我们之前通过不断配置的ABuilder类中字段的值(A, ABuilder字段是一致的);

## 原型模式
原型模式与对象的拷贝息息相关, 原型模式使用 **原型实例** 指定待创建对象的类型, 并通过复制这个原型来来创建新的对象；也就是, 原型作为模板, 通过克隆操作, 来产生更多的对象.

### 深拷贝与浅拷贝
+ **浅拷贝** : **基本类型**数据将值复制给拷贝对象, **引用类型**数据则是将地址复制给拷贝对象, 这基本就是拷贝了个寂寞
+ **深拷贝** : 无论基本类型还是引用类型, 深拷贝会将所有数据拷贝为一个全新的对象

### 实现方式
Java中, 我们可以使用 **cloneable** 接口提供的拷贝机制, 来实现原型模式。implement Cloneable, 重写clone方法。直接使用父类实现, 这只是浅拷贝. 实现深拷贝需要再使用父类实现后, 对各个字段进行赋值, 以达到深拷贝的目的.

## 装饰者模式
对现有的类进行装饰, 核心在于: 在不改变一个类的本身功能的基础上, 为类增加额外的行为, 并且是通过组合的形式完成, 而非通过传统的继承方式

### 实现方式
现有一个普通类A, 创建一个装饰者B, 在B中组合一个A类实例对象, 可以在B的方法中对A的方法进行 **装饰前缀处理**和 **装饰后缀处理**, 甚至装饰者B可以extends A, 用于套娃装饰(不推荐)

## 代理模式
代理模式与装饰者模式很相似, 但请记住: 当无法直接访问某个对象或访问某个对象有困难时, 我们就可以通过一个代理对象进行间接访问

### 实现方式
现有一个无法直接访问的类A, 我们需要为其创建一个代理类B, 在B中组合一个A类对象, 可以在B的方法中对A的方法进行 **代理前绕处理**和 **代理后绕处理。**

这么一看, 装饰者模式和代理模式并无区别, 在实现方式上, 两者并无太大的区别，两者只是侧重点不同。  
装饰者模式侧重于:  增强自身。代理模式侧重于:  别人帮你做事, 也可以添加一些于业务本身并无太大关系的业务。
