# JUC概述


## 什么是JUC


> JUC: java.util.concurrent，这是Java的一个处理线程的工具包，JDK1.5开始出现。
>



## 进程和线程


### 进程、线程


> **进程（process）**：是计算机程序关于某数据集合的一次运行活动，是计算机系统进行资源分配和调度的基本单位，是操作系统结构的基础。  
**线程（thread）**：是操作系统能够进行运算调度的最小单位。被包含在进程中，是进程中实际运行的单位。
>



### 线程状态


**线程状态枚举类 Thread.State**



```java
public enum State{
	NEW, // 新建
	RUNNABLE, // 准备就绪
	BLOCKED, // 阻塞
	WAITING, // 等待中（不见不散）
	TIMED_WAITING, // 等待中（过时不候）
	TERMINATED, // 结束
}
```



### wait()与sleep()


1. wait()是Object的方法，任何实例对象都能调用。sleep()是Thraead的静态方法。
2. wait()会释放锁，但调用它的前提是当前线程占有锁（即代码synchronized中）。sleep()不会释放锁，它也不需要占有锁。
3. wait()与sleep()都会被interrupted()中断。



### 并发与并行的区别


#### 串行/并行


> 串行：多个任务顺序执行  
并行：多个任务同时执行
>



#### 并发/并行


> 并发：是针对单核CPU提出的，CPU在同一时刻只能执行单个任务，但CPU在多个任务间不断切换执行，给人类造成了**多个任务同时执行的假象**。（人脑反应不过来，以为是多个任务同时执行，这是人们为了提高单核CPU的使用效率而做的改进）  
并行：是针对多核CPU提出的，多核 CPU 的每个核心都可以独立地执行一个任务，而且多个核心之间不会相互干扰。多核上的每一个任务才是真正的同时运行。
>



### 用户线程和守护线程


+ 用户(User)线程：运行在前台，执行具体的任务，如程序的主线程、连接网络的子线程等都是用户线程。当主线程结束，用户线程依旧运行时，JVM存活。
+ 守护(Daemon)线程：行在后台，为其他前台线程服务.也可以说守护线程是 JVM 中非守护线程的 **佣人**。一旦所有用户线程都结束运行，守护线程会随JVM一起结束工作。



## 管程


### 临界资源


+ 一次只允许一个进程访问的资源
+ 多个进程只能互斥访问的资源



### 管程的定义


> 管程就是代表共享资源的数据结构以及由对该共享数据结构实施操作的一组过程所组成的资源管理程序共同构成的一个操作系统的资源管理模块。
>



### 管程的作用


> 临界资源的访问需要同步操作，如信号量就是一种方便有效的进程同步机制。但信号量的方式要求每个访问临界资源的进程都具有**wait操作(P操作，申请资源)** **signal操作(V操作，释放资源)**。这样使大量的同步操作分散在各个进程中，不仅给系统管理带来了麻烦，而且会因同步操作的使用不当导致死锁。管程就是为了解决这样的问题而产生的。  
管程被请求和释放临界资源的进程所调用。  
管程定义了一个数据结构和能为并发进程所执行（在该数据结构上）的一组操作，这组操作能同步进程和改变管程中的数据。
>



# Lock接口


## synchronized关键字


> synchronized关键字是一种同步锁
>



synchronized修饰的对象有：



1. 代码块，被修饰的代码块称为同步语句块，作用对象是调用此代码块的对象。
2. 方法，被修饰的方法称为同步方法，作用对象是调用这个方法的对象。
3. 静态方法，作用对象是类的所有实例对象。
4. 类，作用对象是类的所有实例对象。



+ 注意synchronized关键字无法被继承！！！



## 什么是Lock接口？


java.util.concurrent.locks中有两个重要的锁的根接口：**Lock**、**ReadWriteLock**。  
Lock接口实现提供了比使用synchronized更广泛的锁操作。



**Lock与synchronized的区别**



+ Lock不是Java内置的，而synchronized是Java的内置关键字，Lock是一个接口，可以通过这个接口实现同步访问。
+ Lock需要手动释放锁，而synchronized是自动释放锁。当程序发生异常时，synchronized会自动释放锁，而Lock不会自动释放锁，可能导致死锁，所以我们通常在finally{}中进行释放锁操作。
+ Lock可以让等待锁的线程响应中断，而使用synchronized时，等待的线程会一直等待下去，无法响应中断。
+ 使用Lock可以知道是否成功获得锁，而synchronized不行。
+ Lock可以提高多线程的读操作的效率。
+ 性能上：若竞争资源不激烈，两者性能没啥差别；若竞争资源激烈，Lock的性能要远远优于synchronized。



## 线程间的通信


Object中有：



+ wait() 线程等待
+ 通知：notify() 唤醒指定线程、notifyAll() 唤醒其他所有线程



Lock中有newCondition()，创建一个Condition对象，而Condition对象有：



+ await() 线程等待
+ 信号：signal() 唤醒线程、signalAll() 唤醒其他所有线程



### wait()的虚假唤醒


> wait()的特点：在哪里等待，就在哪里唤醒，然后线程继续执行余下代码。
>



正是由于wait()有这样一个特点，当线程已经处于等待状态中，但此线程又抢到了CPU，那么此线程就会被唤醒然后继续执行，就造成了线程的虚假唤醒。



**解决方法**  
wait()应当始终在**循环**中使用，以保证线程被虚假唤醒时依旧需要进行循环条件的判断。这样的话，线程如果符合循环条件，就依旧会执行wait()，保持等待状态。
