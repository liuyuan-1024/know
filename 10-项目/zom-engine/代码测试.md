# Context & Architecture
本项目是一个纯粹的单机底层文本编辑引擎底座，为上层宿主提供内存级别的无头（Headless）纯文本编辑服务。
- 绝对边界：无 UI、无渲染、无 I/O、无原生事件。只有纯粹的数据流。
- 架构核心：绝对字节偏移量坐标系（usize）、CQRS/Delta 驱动、零拷贝极速状态机。
- 测试架构：采用标准集成测试（Integration Tests）模式。所有测试代码统一存放在与 `src/` 同级的 `tests/` 目录下。

# Core Rules: 极简极严苛的测试生成策略

## 1. 测试哲学：纯粹的黑盒状态代数
- 核心模式：所有测试必须严格遵循 `[初始状态] + [Command/Delta] -> [断言目标状态]` 的演进逻辑。
- 黑盒契约：因为运行在 `tests/` 目录，必须将被测引擎视为外部 Crate，**仅允许调用 Public API**。绝不测试内部私有实现，只验证暴露给宿主的纯粹机器契约。
- 拒绝废话：**绝对禁止**在测试代码中使用 `// Arrange`, `// Act`, `// Assert` 或 `// 初始化` 等常规废话注释。代码的结构本身就必须自解释。

## 2. 边界击穿 (Extreme Boundary Testing)
- 坐标系刺探：凡是涉及 `usize` 偏移量的逻辑，强制生成针对 UTF-8 字符内部截断、Grapheme Cluster 边界分割、越界偏移量（Out-of-bounds）的极端测试用例。
- 契约验证：如果原 API 设计为 Panic，必须使用 `#[should_panic(expected = "...")]` 精准断言；如果是 Result，必须断言到具体的 Error Variant。

## 3. 结构压缩与信噪比
- 矩阵驱动：当需要验证同一行为的多个边界值时，强制使用数组迭代（Table-driven tests）或极简宏来压缩代码，绝对禁止复制粘贴产生大量相似的单一测试函数。
- 命名规约：测试函数名必须像契约一样清晰，采用 `action_condition_expected_result` 蛇形命名法（如：`apply_insert_at_invalid_utf8_boundary_should_panic`），杜绝含糊不清的命名。

# Vocabulary Constraints (🚫 词汇禁区)
- 严禁出现："Simulate Click", "Mock Window", "Render Result" 等任何宿主端/UI端词汇。
- 强制使用："Dispatch Command", "State Transition", "Apply Delta", "Byte Offset", "Boundary" 等领域词汇。
